Empty state messages

The purpose of empty state messages

Empty state messages play an important role in creating the user’s first impression of a website, app, or service. If you're ever wondering why so many users quit within the first 3 days after installing an application, here’s the answer. Empty states are UI screens or components that show up when there’s no content or information to display. That may happen when you first enter an app after downloading it or when you’ve just deleted all data associated with a certain feature and there’s nothing left to show.

What’s important is that empty state messages are useful across the whole application, without looking at how advanced a particular user is.

Empty states can appear within a specific feature or as a full-screen message. When writing an empty state message, remember that most users scan rather than read. So it’s important to skip unnecessary details. You can also include images or visual cues to enhance the user experience.

Empty state messages should explain why the space is empty and suggest what actions the user should take to move forward. What’s more, it’s also a good way to show brand tone and empathy.

The most common types of empty states are:

  • First-time use - when the user just enters an application and there’s no data added yet.

  • No results - when the user searches for something and the are no records to show.

  • Error - when content fails to load or something goes wrong.

  • Cleared content - when the user deletes everything or completes an action and there’s nothing else left to display.

Examples of empty state messages

Google Currents displays the following empty state message when the Communities tab is empty. There’s a small image above the message, and a short description informing that no communities have been created so far. It also gives the possibility to create a new community or look for some inspiration by clicking on the link. This empty state should satisfy everybody. It’s clear and informative but not too descriptive.

Another example comes from Slack and is displayed when there are no search results found. The searched term is repeated for clarity, and Slack suggests possible reasons for the lack of results. If the user still isn’t satisfied, they’re given an option to provide feedback.

Amazon’s empty state is similar to Google Current’s. It clearly communicates that there are no albums yet. They also provide a hint on how albums can be used. Moreover, there’s a CTA on the button so a user can just start by creating a new album right away.

Recommendations and best practices

Writing good-quality empty state messages can help the writer and users in many ways. First of all, the user stays informed and knows what to do. Avoid saying that there’s nothing here yet. Instead, you should write what is supposed to be here or what the user can do. If relevant, feel free to add short instructions so that users are informed how to start using a feature. Another important aspect is that thanks to well-crafted empty state messages, you can build a good relationship between the user and your product. And that is worth more than anything else. Don’t be afraid to use pictures or any visual cues. These elements are really great to focus the user’s attention. If possible, include CTA on buttons so that users don’t have to look elsewhere to start using a feature.

The table below presents recommendations and best practices for the key aspects of empty state messages.

Empty state messages Recommendations and best practices
Title

  • Write short titles and make them scannable.

  • Write in sentence case and don’t use punctuation (unless it’s a question).

  • Write titles that will attract users to start using a feature.
Body

  • Inform what’s the reason for the empty state. If users can't do anything else, celebrate completing a task.

  • Don’t write more than 2 sentences.

  • Avoid too technical information that could discourage users from discovering a feature.
Call to action (CTA)

  • A CTA button should always complement the title of the empty state.

  • If applicable, give users the possibility to dismiss or cancel so they don’t feel pressured to continue.

  • Use as few words for CTA as possible.
Accessibility

  • Don’t use jargon.

  • If there are any links needed, make them as descriptive as possible.

  • Make text easily scannable to focus the user’s attention on key information.
