Error messages

The purpose of error messages

Things don’t always run smoothly. Software can fail, and users may also make mistakes. Error messages play an important role in the user experience.

Error messages appear to inform users that something went wrong and they’re probably unable to complete a task. The message should draw attention and be very specific about what has just happened and what users need to do to move forward. They may even provide a way forward in the call to action.

Preparing error messages is not an easy task to do. They must be clear and precise, reporting what happened without unnecessary details. Remember, the more generic an error message is, the less helpful it becomes.

Error messages need to convey the following:

  • Explain simply that an issue occurred and what that issue is about.

  • Provide a solution to that issue so users aren’t left alone and they know what to do to complete the process or correct their mistakes.

  • Turn the delay into an experience that is as pleasant as possible.

Error messages shouldn’t include any jargon or phrases that users won’t be familiar with. Keep messages as simple as possible to avoid making users more frustrated. If many possible situations can result in getting the same error message, consider redirecting people to a documentation page where they’ll find all possible scenarios and solutions. There’s nothing worse than displaying an error message that is too general and confuses people even more.

Examples of error messages

Many error messages aren’t helpful because they say what’s impossible instead of what is possible. Let’s take a look at the following examples.

Incorrect Correct Explanation
Password is invalid. Password should be 12 digits. Be specific. There might be several reasons why an entered password is not valid. Users need to know what exactly is wrong.
You can’t book this flight. Direct flights to London are available only in June. People don’t want to wonder what’s going on and why they can’t make a reservation. They want to know why they can’t book a flight.
Sorry, but your username doesn’t exist. Wrong username. Check for typos and try again. Giving hints in error messages is recommended. Instead of making people scared that their username doesn’t exist, encourage them to double-check that the username is correct.

Recommendations and best practices

The examples above confirm that companies have various approaches. When writing error messages, make sure that they are straightforward to understand and don’t confuse or frustrate users. If a user needs to take action to resolve a situation, be clear and specific about that. If a problem is more complex, consider preparing a documentation page and redirecting users there. Avoid excessive details unless they’re really important.

The table below presents recommendations and best practices for the key aspects of error messages.

Error messages Recommendations and best practices
Title

  • Use titles only when needed.

  • Avoid explaining what to do in titles.

  • Avoid using exclamations.
Body

  • If it’s not possible to detect what situation resulted in displaying an error, don’t make up any reasons.

  • Don’t write more than 2 sentences.

  • Avoid too technical information in the message.
Call to action (CTA)

  • Be specific. If an error message invokes a choice, use imperative verbs such as “Save” or “Create”.

  • Use as few words for CTA as possible.
Accessibility

  • Don’t use jargon.

  • If there are any links needed, make them as descriptive as possible.

  • Make text easily scannable to focus the user’s attention on key information.
